## Problem Statement

Data scientists in search-heavy engineering orgs face a dual challenge when brainstorming analyses:
1. **Analytical rigor gap** — no one challenges their methodology before they invest days in execution. By the time a peer reviews, it's too late to pivot.
2. **Stakeholder framing gap** — the analysis is technically sound but doesn't land with business execs because it's framed around methodology, not business outcomes. Search is eng-heavy; translating DS insights for non-technical audiences is consistently undervalued.

Today's workaround: manually prompting Claude with an IC9 persona to debate methodology. No audience intelligence, no structured output, no grounded understanding of what stakeholders actually care about.

## What Makes This Cool

A Socratic debate partner that simultaneously optimizes for analytical rigor AND stakeholder impact. Like how content creators reverse-engineer viral posts to understand what audiences care about, this tool reverse-engineers what business execs care about and helps you frame your DS analysis to maximize influence. Three independent perspectives (methodology, stakeholder, pragmatism) debate your analysis plan before you spend a single hour executing it.

## Constraints

- **Skills/subagents in their own repo** — not a web app. Deployable as Claude Code skills.
- **Modified hybrid architecture** — 3 subagents with SHARED search context (one search, 3 perspectives). Not 3 independent searches.
- **Must produce structured output** — idea 2 (report generator) will consume this downstream.
- **Socratic style** — debate, not checklist. Back-and-forth, not one-shot evaluation.
- **User is learning** — building multi-agent architecture for the first time in a brainstorming context (vs. ds-review's evaluation context).
- **Idea 3 deferred** — the full pipeline is v2. This is a standalone module.
- **v1 audience signals are generic** — web search returns industry-level context, not company-specific exec priorities. Company-specific intelligence requires user-provided context about their stakeholders. The search grounds the discussion; the user grounds the specifics.
- **Effort is L, not M-L** — 3 persona files, search grounding, SKILL.md orchestration with multi-agent dispatch, output contract, AND evals is a substantial build for a first multi-agent brainstorm skill. Setting expectations honestly.

## Premises

1. Ideas 1+2 are modular components in their own repo, not embedded in existing projects.
2. Idea 1 (Socratic brainstorm) is higher-priority than idea 2 (report generator).
3. Idea 3 (mini DS Dr. Claw) is deferred to v2.
4. The "audience intelligence" feature (reverse-engineering what execs care about) is the unique differentiator.
5. One shared search step grounds all 3 personas (not 3 independent searches).

## Approaches Considered

### Approach A: Socratic Skill (minimal)
Single SKILL.md with rich prompt. No subagents, no search. Pure prompt engineering.
Effort: S | Risk: Low | Completeness: 6/10
- Pros: Ships fastest, easy to iterate
- Cons: No grounded audience intelligence, no structured output

### Approach B: Debate + Ground (single agent + search)
One skill with Socratic debate + search-powered audience intelligence + structured output.
Effort: M | Risk: Medium | Completeness: 8/10
- Pros: Unique differentiator built, structured output for idea 2
- Cons: Single perspective limits depth

### Approach C: Modified Hybrid (CHOSEN)
3 subagents with shared search context. One search grounds all 3 perspectives.
Effort: L (human: ~2 weeks / CC: ~5-6 hours) | Risk: Medium | Completeness: 8.5/10
- Pros: Multi-persona architecture, reusable subagents, richest debate quality
- Cons: More complex coordination, unvalidated persona definitions (need iteration)

## Recommended Approach: Modified Hybrid

### Architecture

```
User Input (analysis question/plan)
    |
    v
[Phase 0: Scope Check + Search Grounding]
    |-- If input contains multiple analysis questions, surface them and ask user
    |   which to brainstorm first. One question per session.
    |-- Read project CLAUDE.md for domain context
    |-- Search for audience signals (web search, trending topics, exec priorities)
    |-- Search for prior analyses (if knowledge base exists)
    |-- Assemble shared evidence block as markdown section
    |-- FALLBACK: If search returns nothing relevant, proceed with user-provided
    |   context only. Flag to user that audience signals are ungrounded.
    |
    v
[Phase 1: Three Perspectives (sequential in v1, parallel in v1.1)]
    |-- NOTE: v1 runs subagents sequentially (~30-60s each) to simplify
    |   SKILL.md orchestration. Switch to parallel dispatch in v1.1 once
    |   the sequential version works.
    |
    |-- [Methodology Critic]     -- IC9-style rigor. Challenges assumptions,
    |   |                           statistical methods, data quality, confounders.
    |   |                           Lens: "Is this analysis SOUND?"
    |   |
    |-- [Stakeholder Advocate]   -- Business exec perspective. What metrics matter?
    |   |                           What story lands? What gets funded?
    |   |                           Lens: "Will this analysis INFLUENCE decisions?"
    |   |
    |-- [Pragmatist]             -- Data availability, timeline, feasibility.
    |                               What can you actually do with what you have?
    |                               Lens: "Can this analysis SHIP?"
    |
    v
[Phase 2: Cross-Persona Challenge Synthesis]
    |-- Synthesizer agent (orchestrator, not a subagent) reads all 3 outputs
    |-- Stages disagreements as explicit exchanges:
    |   "Methodology Critic says X. But Pragmatist counters: Y."
    |-- Surfaces agreements (all 3 align = strong signal)
    |-- Surfaces tensions as structured back-and-forth, not summary
    |-- Present to user as multi-perspective challenge, not consensus mush
    |
    v
[Phase 3: Socratic Dialogue Loop]
    |-- DRIVER: The orchestrator SKILL.md drives the loop (not a subagent)
    |-- It uses a unified "challenger" voice that draws on all 3 persona outputs
    |-- FLOW per round:
    |   1. User responds to the challenge synthesis
    |   2. Orchestrator identifies which persona's concerns the user addressed/ignored
    |   3. Orchestrator pushes back on unaddressed concerns ("You addressed the
    |      methodology question, but the Stakeholder Advocate's framing concern
    |      is still open: how would you explain this to a VP in 30 seconds?")
    |   4. Orchestrator may invoke a specific subagent for deeper follow-up
    |      if the user's response opens a new thread
    |-- TERMINATION: User says "enough" / "I'm good" / "let's move on",
    |   OR 3 rounds complete (configurable MAX_DEBATE_ROUNDS), OR convergence
    |   (user has addressed all major tensions)
    |-- SUBAGENT RE-INVOCATION: Subsequent rounds do NOT re-invoke all 3 subagents.
    |   The orchestrator uses persona outputs from Phase 1 as reference. Only
    |   re-invokes a specific subagent if user's response fundamentally changes
    |   the analysis plan (e.g., "actually, I'll use a completely different method").
    |-- STRUCTURED CAPTURE: Each round is captured in dialogue_history with
    |   role = "user" or "challenger" and a `references_persona` field
    |
    v
[Phase 4: Structured Output]
    |-- Human-readable: Brainstorm summary (narrative, key decisions, open questions)
    |-- Machine-readable: JSON with claims, evidence, methodology choices, stakeholder framing
    |-- Feed downstream: idea 2 (report generator) consumes the JSON contract
```

### Subagent Evidence Stratification (shared search, different framing)

All 3 subagents receive the SAME search results and evidence block, injected as a literal markdown section in each subagent's dispatch prompt. Each gets a different attention directive as a system-level instruction block in their .md reference file (borrowed from the ds-review design):

| Subagent | Attention Directive |
|----------|-------------------|
| Methodology Critic | "Focus on: statistical claims, methodology choices, data quality signals, potential confounders. Use search results to find similar analyses that succeeded or failed." |
| Stakeholder Advocate | "Focus on: business context signals, exec priorities from search results, metric definitions that align with business outcomes, narrative framing." |
| Pragmatist | "Focus on: data availability signals, timeline constraints, tool/infrastructure limitations, scope that's achievable." |

### Persona Definitions (v1, expect iteration)

**Methodology Critic (IC9-style):**
- Senior DS with 10+ years experience
- Skeptical of shortcuts, demands statistical rigor
- Key questions: "What's your null hypothesis?", "What confounders have you considered?", "Would this pass peer review?"
- Calibrated severity: doesn't kill ideas, but demands they're well-founded

**Stakeholder Advocate:**
- Product-minded DS who's given 100+ exec presentations
- Thinks in terms of: "So what?", "Why now?", "What should we DO differently?"
- Key questions: "If this analysis proves your hypothesis, what decision changes?", "Can you explain this to a VP in 30 seconds?"
- Draws on search results to ground what audiences actually care about

**Pragmatist:**
- Senior IC who's shipped dozens of analyses
- Thinks in terms of: timelines, data quality, tool availability
- Key questions: "Do you actually have this data?", "How long will this take?", "What's the 80/20 version?"
- Catches scope creep and data fantasies

### Domain Knowledge Architecture (domain-agnostic core, domain-aware when configured)

The brainstorm agent is domain-agnostic by default — it works for any DS analysis question. But domain-specific knowledge dramatically improves framing quality.

**Pattern:** Same as ds-review's `--domain` flag. Without domain context, all 3 personas give general advice. With domain context, they ground their challenges in domain-specific knowledge.

**How it works:**
```
ds-brainstorm --domain search-relevance
```

1. **Domain knowledge files** live in `references/domains/`:
   ```
   references/
   ├── domains/
   │   ├── search-relevance.md    # NDCG, MRR, query understanding, ranking signals
   │   ├── experimentation.md     # A/B testing, switchback, interleaving
   │   ├── recommendation.md      # CTR, engagement, diversity-relevance tradeoff
   │   └── (extensible)
   │
   ├── methodology-critic.md
   ├── stakeholder-advocate.md
   └── pragmatist.md
   ```

2. **Domain file gets injected into each persona's evidence block** alongside the shared search results. Each persona uses it differently:
   - **Methodology Critic:** "For search relevance, are you using NDCG or MRR? Have you accounted for position bias?"
   - **Stakeholder Advocate:** "Search execs care about query success rate and abandonment, not NDCG. Frame your findings in terms they track."
   - **Pragmatist:** "Do you have click logs with position data? Without that, offline NDCG is unreliable."

3. **Without `--domain`:** Personas give general DS advice. Still useful, just less targeted.

4. **Knowledge source for search-relevance domain:**

   **Primary seed:** `ds-productivity-agents/dev/test-fixtures/real/atlassian-rovo-search-relevance.md` (102 lines). This is a real-world search relevance analysis covering Atlassian Rovo's search architecture. Copy this file into the new repo as the seed for `references/domains/search-relevance.md`.

   **Critical search context this provides (what makes brainstorming search-aware):**
   - **Architecture vocabulary:** L1 retrieval (BM25 + KNN), L2 reranking (cross-encoder + DCN), L3 interleaving — personas should use these terms
   - **Evaluation metrics:** QSR (north-star online), NDCG/MRR/Recall@k (offline), LLM-as-Judge — Methodology Critic should challenge metric choices using this vocabulary
   - **Personalization signals:** user affinity, contributor authority, freshness — Stakeholder Advocate knows what execs track
   - **Query understanding:** intent classification, query rewriting, typo correction, acronym handling — domain-specific methodology concerns
   - **Multi-stack strategy:** document vs messaging vs default search — Pragmatist knows what data is available where

   **Supplementary sources:** NotebookLM search relevance notebook, `search-domain-knowledge` skill digests, user's Maven course notes. These enrich the seed over time.

   **For non-search domains:** Start with an empty domain file and populate as needed. The architecture doesn't depend on the domain file being rich — it just makes the brainstorm more targeted when it is.

**Why this matters for search:** Search is eng-heavy. The right framing for a search analysis is NOT the same as the right framing for a churn analysis. A domain-agnostic brainstorm agent would suggest generic "build a dashboard" advice. A search-aware one would suggest "compare NDCG@10 lift against query abandonment rate and frame it as user success."

### Search Grounding (Phase 0)

One search step that gathers:
1. **Domain context** — web search for recent developments in the user's analysis domain
2. **Audience signals** — what business leaders are talking about in this space (optional: twitter-cli, web search for "[company/industry] CEO priorities")
3. **Prior art** — similar analyses that have been done before (prevents reinventing the wheel)

Search results are deduplicated and assembled into a shared evidence block. All 3 subagents receive the same block with different attention directives.

## Success Criteria

1. 3 subagents produce meaningfully different perspectives on the same analysis question (not 3 variations of the same advice)
2. Search grounding surfaces at least 1 relevant signal the user hadn't considered
3. Synthesis correctly identifies tensions between subagents (not consensus mush)
4. Socratic dialogue pushes back on user responses at least once per round
5. Structured output JSON is parseable and contains all contract fields
6. Eval test cases: ≥3 test brainstorm questions. v1 evals are structural (all 3 perspectives present, JSON parseable, tensions array non-empty, each persona mentions at least 1 domain-specific concern). Quality evals (did the Critic challenge the RIGHT assumptions?) deferred to v1.1 after manual validation establishes what "right" looks like.
